# 计算机视觉岗-要点学习

## 1. 这份文档怎么用 √

这份文档是一份偏详细的知识手册，目标不是把所有理论都推到最深，而是服务于下面三件事：

- 面试时能解释清楚常见概念。
- 做项目时能判断问题大概出在哪里。
- 遇到追问时不只会说“我跑通过”。

阅读方式建议：

1. 先看和自己项目最相关的章节。
2. 每看完一块，就回到自己做过的项目里找对应例子。
3. 尝试不用资料，自己复述一遍。

## 2. 深度学习基础 √

### 2.1 前向传播、损失函数、反向传播

前向传播就是把输入送进网络，经过一层层计算，得到预测结果。  
损失函数用来衡量预测结果和真实标签差了多少。  
反向传播则是把这个误差往回传，算出每个参数应该朝哪个方向更新。  
优化器根据梯度更新参数，常见的有 `SGD` 和 `Adam`。

可以这样理解：

- 前向传播：模型“做题”
- 损失函数：老师“判卷”
- 反向传播：告诉模型“错在哪”
- 优化器：让模型“按反馈改正”

如果这几个环节中任一环节有问题，训练就容易异常，比如：

- 标签错了，损失就失真。
- 学习率太大，参数更新会来回震荡。
- 学习率太小，loss 会降得很慢。

### 2.2 学习率、batch size、epoch、iteration

`learning rate` 决定每次参数更新的步长。  
`batch size` 是每次送入模型训练的样本数量。  
`iteration` 是一次参数更新，模型看完一个batch数据做一次返校传播和参数更新。  iteration = 数据集数量 / batch_size
`epoch` 是整份训练集被完整遍历一遍。

常见现象：

- 学习率太大：loss 波动大，不稳定，甚至直接发散。
- 学习率太小：loss 会慢慢降，但训练很慢，可能卡在较差解附近。
- batch size 太小：梯度噪声更大，训练不稳定，但有时泛化更好。
- batch size 太大：训练更稳定，但显存压力更大，泛化不一定更好。

### 2.3 过拟合、欠拟合、正则化

欠拟合表示模型太弱，训练集都学不好。  
过拟合表示模型在训练集表现很好，但在验证集或测试集泛化差。

常见判断：

- 训练集和验证集都差：更像欠拟合。
- 训练集好、验证集差：更像过拟合。

缓解过拟合的常见方法：（凡是能抑制过拟合、提升泛化的训练约束或策略，广义上都可以叫正则化）
- 增加数据量（数据层面样本扩充，减少对固定数据分布的死记硬背）
- 做更合理的数据增强
- 降低模型复杂度
- 使用L1/L2正则化 （参数约束，狭义正则化）
- 使用 `Dropout`（网络层面，训练时随机屏蔽部分神经元）
- 提前停止训练（训练策略层面）

### 2.4 BatchNorm 和 Dropout

`BatchNorm` 主要用于稳定训练、加速收敛，它会对中间特征做标准化处理。  
`Dropout` 是训练时随机“丢掉”一部分神经元，防止模型过度依赖某些局部模式。

直观理解：

- `BatchNorm` 更像是在帮训练过程“走稳”
- `Dropout` 更像是在防止模型“死记硬背”

### 2.5 面试里至少要能说清的内容

- 为什么训练不收敛？
- 为什么学习率很关键？
- 为什么会过拟合？
- `BatchNorm` 和 `Dropout` 分别解决什么问题？

如果只能背一句定义，通常不够。最好能结合项目说：

- 我看到验证集波动大，先排查学习率和 batch size。
- 我发现训练集指标很高而验证集不高，优先怀疑过拟合或数据分布问题。

## 3. 计算机视觉基础任务与指标

### 3.1 图像分类、目标检测、分割、关键点、跟踪、ReID 的区别

图像分类只回答“这张图是什么”。  
目标检测回答“图里有什么，以及它在哪里”。  
语义分割回答“每个像素属于哪一类”。  
实例分割在语义分割基础上进一步区分不同个体。  
关键点定位关注目标上的重要点位，比如人体关节点。  
多目标跟踪处理视频中的连续目标关联问题。  
ReID 主要解决“这个目标是不是之前那个目标”的身份识别问题。

最容易混淆的点：

- 检测是“找框 + 分类”
- 跟踪是“跨帧保持同一目标身份”
- ReID 是“提供跨时间或跨视角的外观身份依据”

### 3.2 Precision、Recall、F1

`Precision` 关注“你说是目标的里面，有多少真的是目标”。  
`Recall` 关注“真实存在的目标里，你找出来多少”。  
`F1` 是两者的综合。

简化理解：

- Precision 低：误检偏多
- Recall 低：漏检偏多

### 3.3 IoU、AP、mAP

`IoU` 是预测框和真实框的重叠程度。  
如果预测框和真实框几乎重合，IoU 就高；重合差，IoU 就低。

`AP` 可以理解为某一类别在不同阈值下综合表现的分数。  
`mAP` 是多个类别 AP 的平均值，是检测任务里最常见的指标之一。

你不一定要在面试里推公式，但至少要知道：

- `mAP` 不是只看分类对不对，还看框得准不准。
- 仅仅多检出几个框，不一定会让 `mAP` 提高。

### 3.4 跟踪指标和 ReID 指标

多目标跟踪常见指标有：

- `MOTA`
- `IDF1`

其中 `IDF1` 更能反映身份保持得稳不稳。

ReID 常见指标有：

- `Rank-1`
- `Rank-5`
- `mAP`

最关键的理解是：

- 检测任务重在“找得到、框得准”
- 跟踪任务重在“身份关联稳不稳”

## 4. 目标检测

### 4.1 目标检测的基本流程

典型流程包括：

1. 输入图像
2. 提取特征
3. 预测候选框和类别
4. 过滤低质量结果
5. 使用 `NMS` 去重
6. 输出最终检测框

面试里不需要把所有实现细节背下来，但要知道模型做的事大概是什么。

### 4.2 one-stage 和 two-stage

two-stage 检测器通常先生成候选区域，再对候选区域分类和回归，例如 Faster R-CNN。  
one-stage 检测器通常直接在特征图上完成类别和位置预测，例如 YOLO 系列。

一般来说：

- two-stage 往往精度强
- one-stage 往往速度快、工程部署更友好

所以如果岗位强调实时性，`YOLO` 之类模型更常见。

### 4.3 YOLO 为什么快

因为它把检测任务做得比较“直接”：

- 不先单独生成候选区域
- 直接在特征图上预测框和类别
- 整体流程更适合端到端推理

这不代表它永远最优，而是说明它在速度和工程落地方面通常有优势。

### 4.4 Anchor 和 Anchor-free

`anchor` 可以理解为预先设定的一组候选框模板。  
模型在这些模板基础上做偏移调整。  
anchor-free 则不再强依赖预设框模板，而是直接预测目标位置。

理解重点不是背实现，而是知道：

- anchor 机制帮助模型从预设框出发学习目标位置
- anchor-free 可以减少一部分手工设计和超参依赖

### 4.5 NMS 是什么

`NMS` 是非极大值抑制。  
当模型对同一个目标输出多个相似框时，`NMS` 会保留最可信的框，去掉高度重叠的重复框。

如果 `NMS` 设置不合理，可能出现：

- 框太多，误检看起来更多
- 框被抑制过度，漏检增加

### 4.6 小目标和复杂场景为什么难

小目标通常像素少、特征弱、容易被背景淹没。  
复杂场景常见问题包括：

- 光照变化
- 遮挡
- 视角变化
- 目标密集
- 背景相似

所以你做交通或隧道场景时，如果夜间效果差，不一定是模型“没用”，也可能是数据覆盖不够。

### 4.7 漏检和误检怎么分析

漏检常见原因：

- 样本覆盖不够
- 小目标太难
- 阈值过高
- 遮挡严重
- 模型表达能力不足

误检常见原因：

- 背景与目标太相似
- 标注边界不一致
- 置信度阈值过低
- 数据噪声较大

一个实用思路是：

1. 先看错误案例长什么样
2. 再判断是场景问题、阈值问题还是训练问题
3. 最后再考虑是否换模型

### 4.8 面试时如何讲检测项目

不要只说“我训练了 YOLOX，mAP 提高了 8%”。  
更完整的表达应该是：

- 场景是什么
- 数据有什么难点
- 我为什么选 YOLOX
- 我做了哪些增强和调参
- 提升体现在哪些指标
- 误检漏检分别有什么变化
- 还有哪些瓶颈没解决

## 5. 多目标跟踪与 ReID

### 5.1 跟踪和检测的关系

多目标跟踪通常以检测结果为基础。  
检测负责每一帧“看到谁”，跟踪负责跨帧判断“这个是不是刚才那个”。

所以检测是跟踪的基础。  
如果检测本身大量漏检，跟踪通常也很难稳定。

### 5.2 卡尔曼滤波和匈牙利算法的直观理解

卡尔曼滤波可以理解为：

- 根据历史状态预测目标下一时刻大概会在哪
- 当当前帧检测到目标后，再把预测和观测融合

匈牙利算法可以理解为：

- 在当前帧检测结果和已有轨迹之间寻找“最优配对”

不要求你背数学推导，但至少要懂它们在跟踪中各负责什么。

### 5.3 DeepSORT 的基本思路

`DeepSORT` 不只看位置，也看外观特征。  
它会综合：

- 运动信息
- 检测框位置
- 外观 embedding

因此在目标交叉、外观差异明显的场景下，它通常比只靠位置关联更稳。

### 5.4 ByteTrack 的基本思路

`ByteTrack` 的关键想法之一是：  
不要把低置信度检测框完全当成垃圾丢掉，而是让它们参与后续关联。

这背后的直觉是：

- 有些低分框其实是真目标，只是当前帧质量不高
- 如果完全丢掉，轨迹会中断

所以 `ByteTrack` 往往能减少一些漏关联，提升跟踪连续性。

### 5.5 DeepSORT 和 ByteTrack 的区别

一个简化理解：

- `DeepSORT` 更强调“外观 + 运动”
- `ByteTrack` 更强调“利用不同置信度检测框做更稳的关联”

这并不意味着谁绝对更强，而是它们擅长的问题侧重点不同。

### 5.6 什么是 ID Switch

`ID Switch` 指系统原本在跟踪同一个目标，后来把它错误地当成了另一个身份。  
这是跟踪任务里很关键的问题，因为它直接反映身份保持能力。

常见原因：

- 遮挡
- 目标交叉
- 检测丢失
- 外观特征不稳定
- 仅靠位置关联不够

### 5.7 跨镜跟踪为什么更难

因为跨镜场景下通常存在：

- 视角变化更大
- 光照差异更大
- 分辨率变化
- 时间不同步
- 同类目标更多

所以跨镜匹配更依赖稳健的 ReID 特征，而不是只看位置。

### 5.8 如何排查跟踪效果差

一个实用排查框架：

1. 先看检测质量是否足够
2. 再看关联逻辑是否合理
3. 再看外观特征是否稳定
4. 最后看阈值、轨迹保留策略是否合适

如果一上来就说“换个跟踪器”，通常说明分析不够细。

## 6. 数据、标注、实验设计与调参

### 6.1 数据不是配角，而是核心

很多视觉项目最终效果差，不是因为模型不够新，而是数据没处理好。  
数据层常见问题包括：

- 样本量不足
- 场景覆盖不全
- 类别不平衡
- 标注不一致
- 数据泄漏

### 6.2 train / val / test 划分

训练集用于学习参数。  
验证集用于选模型、调参。  
测试集用于最终评估。

需要注意：

- 不能一边看测试集结果一边调参数
- 相近时间段、相邻视频帧如果切分不当，容易造成隐性泄漏

### 6.3 标注规范为什么重要

标注不一致会让模型不知道“正确答案到底是什么”。  
例如：

- 有的人把目标框得很紧
- 有的人把周边背景也框进去
- 有的人遮挡目标还标
- 有的人不标

这种不一致会直接影响训练稳定性和最终指标。

### 6.4 类别不平衡怎么处理

如果某一类样本太少，模型可能学得不充分。  
常见处理方法：

- 采样平衡
- 数据增强
- 损失加权
- 补充少数类数据

### 6.5 数据增强不是越多越好

数据增强的本质是模拟真实变化、提高泛化。  
但如果增强太激进，可能会破坏样本分布，反而伤害效果。

需要关注：

- 增强是否符合真实场景
- 增强是否破坏了关键目标特征
- 增强后的标签是否仍然合理

### 6.6 Ablation Study 怎么理解

`ablation study` 就是消融实验，目的是回答：

- 到底是哪一项改动带来了提升？

比如你同时改了数据增强、backbone、学习率、输入分辨率，最后提升了 8%。  
如果没有消融，很难说明哪个因素最关键。

### 6.7 一个成熟的实验分析框架

面对效果不好时，可以按下面顺序排查：

1. 数据有没有问题
2. 标注有没有问题
3. 训练策略有没有问题
4. 模型能力够不够
5. 后处理设定是否合理
6. 部署前后是否存在不一致

这个思路比“盲目试更多模型”更专业。

## 7. PyTorch 训练与调试

### 7.1 最小训练闭环要会什么

至少要知道下面几个组件如何协作：

- `Dataset`
- `DataLoader`
- model
- loss
- optimizer
- scheduler
- train loop
- val loop
- checkpoint

如果这套最小闭环都不熟，面试里很容易被问穿。

### 7.2 Dataset 和 DataLoader

`Dataset` 负责定义“如何读取一条样本”。  
`DataLoader` 负责把样本按 batch 组织起来，支持 shuffle、多进程加载等。

你至少要知道：

- 一条样本返回什么内容
- batch 后 tensor 的 shape 会怎么变化
- 数据和标签如何对应

### 7.3 常见报错和怎么想

维度报错：

- 先看输入 shape
- 再看模型期望 shape
- 再看标签 shape

loss 不下降：

- 看学习率
- 看标签是否对
- 看数据预处理是否对
- 看模型是否真的在更新

验证集效果差：

- 看是否过拟合
- 看 train / val 分布是否差异很大
- 看验证逻辑是否写错

### 7.4 为什么这一块很关键

因为很多“会用现成 repo”的人，一旦改数据、改标签、改输入尺寸、改损失函数，就容易卡住。  
如果你能独立解释训练流程，面试官对你的判断会更好。

## 8. 模型部署与推理优化

### 8.1 为什么需要 ONNX

训练框架和部署框架常常不是同一套。  
`ONNX` 可以理解为一种中间表示格式，方便模型在不同框架和平台之间迁移。

简化理解：

- `PyTorch` 更适合研发训练
- `ONNX` 更适合做跨平台桥梁
- 最终部署平台再把 `ONNX` 转为自己的运行格式

### 8.2 为什么会有算子不兼容

因为不同平台支持的运算集合不同。  
有些训练时能用的操作，部署侧未必支持，或者支持方式不同。

这时就可能需要：

- 改模型结构
- 替换算子
- 调整导出方式
- 重新做精度对齐

### 8.3 部署后精度为什么会下降

常见原因包括：

- 预处理不一致
- 后处理不一致
- 算子替换带来误差
- 数值精度变化
- 输入尺寸变化

所以“训练效果好，部署效果差”并不罕见。

### 8.4 FPS、延迟、吞吐量

- `FPS`：每秒处理多少帧
- 延迟：单次推理从输入到输出需要多久
- 吞吐量：单位时间处理总量

面试里要注意，这三者相关但不完全等价。

### 8.5 推理速度慢怎么分析

可以从这些方向看：

- 模型是不是太大
- 输入分辨率是不是太高
- 前后处理是不是耗时
- 硬件是否真正支持关键算子
- 是否可以量化、裁剪、蒸馏

## 9. 项目复盘与面试表达

### 9.1 面试官真正想听什么

面试官通常不只想知道你“做了什么”，更想知道：

- 你看到了什么问题
- 你是怎么想的
- 你为什么这么做
- 你怎么验证自己的判断

### 9.2 一个实用表达模板

讲项目时可以按下面顺序：

1. 项目任务和场景
2. 难点和瓶颈
3. 方案选择原因
4. 关键实验与改动
5. 指标结果
6. 错误案例与分析
7. 下一步优化方向

### 9.3 需要避免的回答

- “我主要是复现了一套开源代码。”
- “这个模型比较火，所以我就用了。”
- “数据增强加上去以后效果就变好了。”
- “这个细节是 AI 帮我做的，我没太看。”

这些回答的问题不在于它们完全错误，而在于它们无法体现你的判断力。

### 9.4 更好的回答方式

例如：

- “我优先选择 `YOLOX`，是因为项目对实时性有要求，同时它的工程落地和后续部署更友好。”
- “我发现夜间漏检偏多，所以先回查数据分布，补充低照度样本，并针对亮度变化设计增强策略。”
- “跟踪中 ID 切换较多，说明只靠位置关联不够稳，因此我引入外观特征并优化关联策略。”

这种表达能体现你在做技术判断，而不是只是调用工具。

---

## 10. 如何从依赖 AI Agent 过渡到真正理解

### 10.1 AI Agent 可以继续用，但要换一种用法

问题不是不能用 AI，而是不能把 AI 当成“跳过理解”的替代品。

更好的用法是：

- 让 AI 帮你生成最小例子
- 让 AI 帮你解释报错
- 让 AI 帮你列排查思路
- 让 AI 帮你比较不同方案

但最后你自己必须能回答：

- 这段代码在做什么
- 为什么改这里
- 如果出问题该怎么验证

### 10.2 一个实用训练方法

以后每次让 AI 帮你做完一段代码，都补三步：

1. 自己复述这段代码的输入、输出和作用
2. 自己解释它为什么能解决当前问题
3. 自己写下一个可能的失败场景

只要坚持这么做，你会慢慢从“使用者”变成“理解者”。

## 11. 复习时的最低达标标准

如果我要说自己基本具备了计算机视觉岗的核心准备，至少应满足：

- 能清楚区分分类、检测、分割、关键点、跟踪、ReID
- 能解释 `Precision`、`Recall`、`IoU`、`mAP`、`IDF1`
- 能说明 `YOLO` 类模型为什么适合实时检测
- 能解释 `DeepSORT` 和 `ByteTrack` 的核心区别
- 能描述一个视觉项目从数据到部署的完整链路
- 能针对错误案例提出合理排查思路
- 能把自己的项目讲成“问题-分析-实验-结果-改进”

## 12. 最后一句提醒

对计算机视觉岗来说，真正有竞争力的人不只是“把模型跑通的人”，而是“知道为什么这样做、出了问题知道怎么查、效果一般知道怎么改的人”。

如果我后面把自己的检测、跟踪、部署项目按这份手册逐项补透，面试表现会比继续机械堆项目更有提升。
