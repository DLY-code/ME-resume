# 算法岗 Transformer 基础

用途：这份笔记专门补 `Transformer` 的底层基础，目标不是推公式，而是把 `DETR` 里那些最容易让人混乱的概念讲顺：`token`、`query`、`key`、`value`、`encoder`、`decoder`、`object query`、一对一匹配。

## 1. 先建立一个总感觉：Transformer 和 CNN 有什么不一样

`CNN` 比较像“拿一个小窗口在图上扫”，擅长提局部特征；`Transformer` 更像“让不同位置彼此交流信息”，擅长建立全局关系。  
所以很多人第一次学 `Transformer` 会觉得抽象，不是因为它一定更难，而是因为它不再用“卷积核扫图”这种很直观的方式思考，而是改成了“我在这个位置时，应该参考全图哪些位置”的思路。

可以先这样记：

- `CNN`：局部扫描，逐层提特征
- `Transformer`：全局检索，按相关性汇总信息

这个区别很重要，因为后面所有概念，基本都围绕“信息怎么被检索、怎么被汇总”展开。

## 2. token 和位置编码：为什么图像也能喂给 Transformer

`Transformer` 最早是为序列设计的，所以它天生更适合处理“一串东西”。  
但图像是二维的，不是一串字，也不是一句话，所以必须先做一步转换：把图像或特征图改造成一个序列。

这里就会出现 `token`。

`token` 可以先理解成“序列里的一个元素”。在 NLP 里，一个词可以是 token；在视觉里，一块图像 patch，或者特征图上的一个位置，也可以看成 token。  
放到 `DETR` 里，通常是图像先经过 `backbone` 变成特征图，然后把特征图展平，展平后的每个位置特征就是一个 token。

但只把 token 喂进去还不够，因为 Transformer 自己不知道这些 token 原来在图上的哪里。它只知道“有这些特征”，却不知道“这是左上角的特征，还是右下角的特征”。  
所以还要加 `position encoding`，也就是位置编码，让模型知道每个 token 的空间来源。

通俗理解：

- token 像一张张信息卡片
- 位置编码像卡片上的地址
- 没有地址，模型就不知道这些卡片原来在图上哪里

## 3. attention、query、key、value 到底在干什么

`attention` 的核心思想其实不复杂：  
当模型处理某个位置时，它不只看自己，而是去看全局哪些位置和自己更相关，然后按相关性把这些位置的信息拿回来。

这里就会出现 `query`、`key`、`value`。

最容易懂的理解方式是：

- `query`：我现在想找什么
- `key`：我这里有什么标签，可以供别人匹配
- `value`：如果匹配上了，我真正提供什么信息

你可以把它类比成搜索：

- `query` 像搜索词
- `key` 像索引标签
- `value` 像正文内容

比如 query 是“红色轿车”，模型就会拿这个 query 去和很多 key 比较。哪些 key 更像“红色轿车”，对应位置就会拿到更高权重；然后这些位置的 value 会被更多汇总出来，形成新的输出特征。

所以 attention 的底层逻辑可以粗略理解成三步：

1. 用 `query` 去匹配一堆 `key`
2. 算出每个位置的重要程度
3. 按重要程度加权汇总对应的 `value`

你现在不用背公式，先把它理解成“按相关性去全局取信息”就够了。

## 4. self-attention 和 cross-attention：到底是谁看谁

这两个词如果不分清，后面学 `DETR` 会一直乱。

`self-attention` 是“同一组元素内部彼此看彼此”。  
比如一串图像 token，每个 token 都能去看其他 token，决定哪些位置和自己更相关，再把这些相关位置的信息聚合到自己身上。  
这会让每个位置不仅知道“我自己长什么样”，还知道“我和图里其他位置有什么关系”。

`cross-attention` 则不是同一组内部互相看，而是一组东西去看另一组东西。  
也就是说，“我自己是一组查询，我要去读取另一组信息”。

最简记法：

- `self-attention`：自己这组内部互相交流
- `cross-attention`：一组查询去读取另一组信息

放到 `DETR` 里：

- `encoder` 主要做图像 token 和图像 token 之间的 `self-attention`
- `decoder` 主要做 `object query` 和图像 token 之间的 `cross-attention`

## 5. encoder 和 decoder 分别在做什么

理解 `DETR`，最稳的方式就是先把 `encoder` 和 `decoder` 的对象分开。

### 5.1 encoder 处理谁

`encoder` 处理的是图像 token，也就是：

- 图像经过 `backbone` 提出来的特征图
- 展平后的那一串 token
- 再加上位置编码

它的工作不是直接输出“车”或者“框”，而是先把整张图的信息整理好。  
具体说，就是让图像中不同位置彼此交流，让每个 token 都带上更丰富的全局上下文。

通俗理解：

- `encoder` 像先把整张图整理成一份资料库
- 每张卡片不再只是“我这里的局部特征”
- 而是会带上“我和图里其他位置的关系”

所以一句话总结：

- `encoder` 处理的是“图像 token 和图像 token 的关系”

### 5.2 decoder 处理谁

`decoder` 的输入不是原始图像 token，而是一组查询向量。  
在 `DETR` 里，这组查询向量就是 `object query`。

`decoder` 会拿着这组 query 去读取 `encoder` 整理好的图像信息。  
每个 query 都会去问：图里有没有一个目标应该归我负责？如果有，它是什么类别、位置在哪里？

所以一句话总结：

- `decoder` 处理的是“object query 和图像 token 的关系”

## 6. object query 到底是什么

这是 `DETR` 里最关键、也最容易误解的概念。

`object query` 不是：

- 已经分类好的类别
- 从图像里裁出来的候选框
- 传统检测器里的 anchor

`object query` 是：

- 一组可学习向量
- 在模型初始化时随机开始
- 在训练中逐渐学会如何“认领目标”

最通俗的理解方式是：

- 它像一组空的目标席位
- 或一组候选目标槽位

比如模型里固定放了 `100` 个 object query，就可以理解为：  
模型先准备了 100 个空座位，看看这张图里有哪些目标可以坐进这些座位。  
有的座位最后负责一辆车，有的负责一个行人，还有很多座位因为没分到真实目标，就学会输出“空”。

## 7. object query 是怎么来的，它是不是类别

`object query` 不是从图像里直接算出来的。  
它通常是模型里预先定义好的一组可学习参数。

例如：

- 定义 `N=100` 个 query
- 每个 query 是一个 `d` 维向量，比如 `256` 维
- 一开始随机初始化
- 训练时跟着整个模型一起更新

所以它的来源是模型参数，不是人工指定，也不是类别标签。

这点非常重要：

- query 不是“车类 query”
- query 不是“行人类 query”
- query 不是“已经分类好的结果”

更准确地说，query 是“用来生成目标预测的查询向量”。  
训练以后，某些 query 可能更常负责某些类型目标，但这不是人为预先写死的，而是训练中逐渐形成的分工。

## 8. object query 的数量能不能理解成标注目标数量

可以粗略往这个方向理解，但不能简单等同。

更准确地说，`num_queries` 表示的是：

- 模型最多准备多少个目标槽位
- 或者说，模型最多允许自己输出多少个目标候选位

它通常是一个固定超参数，不会因为每张图目标数量不同而临时改变。

为什么不能直接等于当前图片的标注数量：

- 一张图可能有 3 个目标
- 另一张图可能有 10 个目标
- 还有的图可能一个目标都没有

如果每张图都临时按标注数改 query 数量，训练和推理流程都会变得不稳定，所以通常会预先设一个固定值，比如 `100`。

这个值怎么理解最稳：

- 如果一张图里只有 3 个真实目标，那就让其中 3 个 query 去和真实目标匹配
- 剩下的 97 个 query 学习输出空结果

所以它不是“当前图片有几个目标，就设几个 query”，而是“模型提前准备足够多的槽位，去覆盖单张图里可能出现的目标数”。

设置时通常参考：

- 数据集中单张图像可能出现的目标上限
- 任务复杂度
- 计算开销

如果设得太小：

- 图里目标很多时，可能装不下
- 会漏掉目标

如果设得太大：

- 计算量更高
- 会有更多 query 去学习空预测

一句话记忆：

- `num_queries` 不是当前图的标注目标数
- 它更像模型预设的“最大目标槽位数”

## 9. 面试里如果被问“Transformer 的 token 参数怎么设”怎么答

这个问题面试里经常说得不够标准，所以不要一上来就只回答一个数字。  
更稳的做法是先拆开：面试官说的 `token 参数`，可能指的是图像 token 的数量、token 的维度，也可能其实是在问 `object query` 的数量。

在 `DETR` 里，常见可以从三层来答：

- 图像 token 数量
  这通常不是手工单独设一个“token 数”，而是由输入分辨率和 `backbone` 输出特征图尺寸决定。
  例如输入越大、特征图越大，展平后 token 就越多。
  所以图像 token 数量本质上和：
  输入分辨率、下采样倍数、最终特征图高宽有关。

- token 维度
  每个 token 不是一个数，而是一个向量。这个向量长度一般和 Transformer 的 `hidden dim` 或映射通道数有关。
  维度越大，表达能力通常更强，但计算和显存开销也会更高。

- `object query` 数量
  如果面试官其实问的是 `DETR` 里的 query 数量，那它通常由单张图可能出现的目标上限来决定。
  它不是当前图片真实目标数，而是模型预设的最大目标槽位数。

面试时最稳的一版回答可以是：

- “如果是 DETR 里的图像 token，它通常不是单独手工设置的，而是由输入分辨率和 backbone 输出特征图尺寸决定；如果是 token 维度，一般要和 Transformer hidden dim 对齐；如果问的是 object query 数量，它更多取决于单张图中潜在目标数量上限和计算开销。”

如果想更专业一点，也可以先反问一句：

- “您这边是指图像 token 的数量和维度，还是指 DETR 里的 object query 数量？”

这不是逃避问题，反而说明你知道这些概念不是一回事。

## 10. decoder 里的 query 到底在做什么

每个 `object query` 都会去和 `encoder` 输出的图像 token 做 `cross-attention`。  
你可以把这个过程理解成：query 拿着自己的“找目标需求”去看整张图，对不同 token 分配不同注意力权重，然后把最相关位置的信息汇总回来，形成一个新的 query 表示。

这个新的表示再送到后面的预测层，输出三类信息：

- 有没有目标
- 是什么类别
- 框在哪里

所以 query 不是最终结果本身，而是“生成结果的起点”。

如果一个 query 对图中一辆车附近的 token 权重最高，那么它最后就可能学会负责那辆车；如果一个 query 没有找到合适目标，它就会输出空预测或背景。

## 11. 是谁告诉 object query 它该找什么

没有人手工规定：

- 这个 query 负责车
- 那个 query 负责人

一开始所有 `object query` 都只是随机初始化的可学习向量，本身没有明确职责。  
真正让它们形成分工的，是训练过程里的三样东西：

- 真实标注
- 一对一匹配
- 损失函数反向传播

更具体地说：

1. 一张图里有真实目标标注，包括类别和框
2. 模型会输出一组 query 对应的预测结果
3. 训练时用匈牙利匹配，把“预测结果”和“真实目标”做一对一分配
4. 分配上的 query 会根据分类损失和框回归损失去更新
5. 没分配上的 query 则学习输出空预测或背景

经过很多轮训练后，某些 query 会逐渐更擅长负责某类目标或某种模式，但这不是人为写死的，而是训练中自然学出来的。

一句话记忆：

- 没有人手工规定 object query 查什么
- 是“真实标注 + 一对一匹配 + 反向传播”让它逐渐学会该关注什么

## 12. 一个具体例子：一张图里 query 和 GT 怎么配对

假设一张图里只有 3 个真实目标：

- 1 辆车
- 1 个行人
- 1 个自行车

但模型固定有 `100` 个 object query，所以它会输出 100 组预测。  
这 100 组预测里，可能有些像车，有些像行人，有些乱七八糟，还有很多其实什么都不像。

训练时会做一对一匹配，大致可以理解成：

- 从这 100 个预测里，挑出最适合那辆车的 1 个
- 挑出最适合那个行人的 1 个
- 挑出最适合那辆自行车的 1 个
- 这 3 个预测分别和 3 个真实目标配对
- 剩下的 97 个 query 都视为“应该输出空”

然后怎么训练：

- 负责车的那个 query，要把类别学成车、框学准
- 负责行人的那个 query，要把类别学成人、框学准
- 负责自行车的那个 query，要把类别学成自行车、框学准
- 其他没分配上的 query，要学会输出“没有目标”

所以 query 的职责不是训练前写好的，而是训练时通过“谁更适合哪个真实目标”的匹配结果，一步步学出来的。

这也是为什么 `DETR` 能减少重复框：  
因为它在训练阶段就要求“一个真实目标尽量对应一个预测”，而不是像传统检测器那样先给出很多重复候选，最后再靠 `NMS` 去重。

## 13. 为什么 DETR 能减少 anchor 和 NMS

传统检测器通常是：

1. 先铺很多 anchor 或候选框
2. 对这些候选框都做预测
3. 同一个目标常常会出现很多重复框
4. 再用 `NMS` 去重

而 `DETR` 的思路是：

1. 直接用固定数量的 object query 预测一组目标
2. 训练时通过一对一匹配，让一个真实目标尽量只对应一个 query
3. 没匹配上的 query 输出空结果

这样做的结果就是：

- 不需要先设计很多 anchor 模板
- 重复预测天然更少
- 对 `NMS` 的依赖也更低

所以你可以把两者区别记成：

- 传统检测器：先产生很多候选，再筛选去重
- `DETR`：直接预测一组目标，并在训练阶段完成分工

## 14. 从整条链路再看一遍 DETR

现在把前面的概念串起来，`DETR` 可以理解成下面这条链：

1. 图像输入
2. `backbone` 提特征
3. 特征图展平为 token
4. 加位置编码
5. `encoder` 让图像 token 之间建立全局关系
6. `decoder` 让 object query 去读取图像信息
7. 每个 query 输出一个候选目标表示
8. 通过分类层和框回归层得到最终类别和位置

一句话浓缩：

- `backbone` 负责看图
- `encoder` 负责整理整张图的信息
- `decoder` 负责派 query 去认领目标

## 15. 面试快答

- `token`：Transformer 处理的序列元素，在视觉里可以是特征图上的一个位置
- `query/key/value`：query 是我要找什么，key 是可匹配的标签，value 是真正被取出的信息
- `self-attention`：同一组 token 内部彼此交互
- `cross-attention`：一组 query 去读取另一组特征
- `encoder`：处理图像 token 和图像 token 的关系
- `decoder`：处理 object query 和图像 token 的关系
- `object query`：一组可学习的目标查询向量，不是类别、不是 anchor、不是框
- `DETR`：用 object query 直接预测一组目标，因此减少 anchor 和 NMS 这类手工设计
